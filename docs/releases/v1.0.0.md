# Process Monitor & Manager — v1.0.0

> *Codename: Spark*
>
> **Pre-release.** Internal development milestone — not restructured or verified
> against a clean environment, and not publicly distributed.

---

## 📖 Overview

Spark is the first version of Process Monitor & Manager, an OS-course project built to explore process listing, hierarchy inspection, and signal-based process control on Linux, exposed through both a terminal menu and a Zenity GUI.

This milestone focuses on establishing the project's entry-point-plus-modules shape — a CLI script and a GUI script, each delegating to shared module scripts for process info and process control — before introducing search, alerts, live views, or repository tooling. Rather than prioritizing a polished contributor experience, this release concentrates on proving the menu-driven workflow end to end.

Not intended for public distribution in its original form — Spark transforms the idea into a working (if not yet fully wired) piece of software.

---

## 🎯 Objectives

This milestone was developed with the following goals:

* Provide a menu-driven way to inspect running processes (`ps`) and their hierarchy (`pstree`).
* Allow killing, suspending, and resuming a process by PID.
* Mirror the same functionality in a lightweight Zenity GUI.
* Establish a module-per-responsibility script layout for future features to build on.

---

## ✨ Added

### 🏗️ Project Foundation

Initial scaffold: `monitor.sh` (CLI entry point) and `monitor_gui.sh` (Zenity GUI entry point), backed by three module scripts — one for process info, one for process control, and one for CLI input dispatch.

---

### 💻 Command-Line Interface

A menu loop offering Show Process Info (`ps -eo pid,ppid,cmd,%mem,%cpu --sort=-%cpu`), Kill / Suspend / Resume by PID, Show Process Tree (`pstree -p`), and Exit.

---

### 🖥️ Zenity GUI

A radiolist mirroring the CLI menu: Show Process Info, Kill a Process, Suspend a Process, Resume a Process, Exit.

---

## 🏛️ Architecture Decisions

Several architectural principles were established during this milestone.

**Entry point + modules.** `monitor.sh` and `monitor_gui.sh` are thin orchestrators; process-info and process-control logic was intended to live in dedicated module scripts sourced by the CLI entry point, keeping each concern in its own file.

**Local-first, no elevation by default.** Every action operates with the invoking user's own permissions — nothing here requests `sudo` on its own.

**Extensibility.** The module split was intentionally kept simple so that search, alerts, and a live view could be layered on in later milestones without restructuring the CLI or GUI entry points.

---

## ⚠️ Known Limitations

As shipped, several things remain unaddressed:

* The module scripts lived at the repository root without the `.sh` extension and without the `modules/` directory that `monitor.sh` expected — **CLI mode did not actually run.** Fixed in [v2.0.0](v2.0.0.md).
* No search, filtering, alerting, or live/auto-refreshing view.
* No automated tests, CI, or contributor documentation.
* Network I/O and disk I/O are not implemented.

These are addressed progressively in subsequent releases.

---

## 📌 Development Notes

The primary objective of Spark is validating the menu-driven CLI/GUI shape rather than delivering a finished, ready-to-clone tool. Development focused on proving that process listing, tree view, and signal-based control worked as designed — the packaging mistake that broke CLI mode on first run wouldn't surface until someone actually tried to clone and run it, which is exactly what v2.0.0 caught and fixed.
