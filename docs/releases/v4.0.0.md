# Process Monitor & Manager — v4.0.0

> *Codename: Signal*

---

## 📖 Overview

Signal builds on the contributor-ready foundation established in Pulse — it makes the tool's fixed assumptions (a single 50%/20% threshold, CPU-only dashboard ordering, no way to take a snapshot with you) configurable instead, and answers a question the project didn't have a good answer to before: since there's no `.exe` to download for a Bash/Python Linux tool, what *does* a packaged release look like?

Building on Pulse's open-source repository structure, this release also wires up an actual release pipeline — pushing a version tag now produces installable artifacts, the same way pushing a tag on a Windows desktop-app repo produces a downloadable `.exe`.

---

## 🎯 Objectives

This milestone was developed with the following goals:

* Make the CPU/MEM alert and color thresholds configurable without editing source.
* Let the Live Dashboard be sorted by more than just CPU.
* Let a user take a process snapshot with them, not just view it on screen.
* Define and automate a real packaged-release story for a Linux CLI/GUI tool.

---

## ✨ Added

### 🎚️ Configurable Alert & Color Thresholds

`PMM_HIGH_THRESHOLD` and `PMM_MED_THRESHOLD` environment variables override the 50%/20% defaults everywhere they're used — CLI colorizing and alerts, GUI alerts, and the Live Dashboard's color coding — with no code changes required.

```bash
PMM_HIGH_THRESHOLD=80 PMM_MED_THRESHOLD=40 ./monitor.sh
```

---

### 🔀 Interactive Dashboard Sorting

Pressing `o` in the Live Dashboard cycles the sort field through CPU → MEM → PID, re-sorting the visible (and filtered) process list on every refresh.

---

### 📤 CSV Export

A timestamped `process_snapshot_<datetime>.csv` snapshot of the current process table — CLI option 9, the GUI's "Export to CSV" action, or `e` in the Live Dashboard (which exports exactly what's currently filtered and sorted on screen). Command names containing commas or quotes are CSV-escaped correctly.

---

### 📦 Packaged Releases

Two new packaging scripts, both driven by version number rather than hardcoded paths:

- **`scripts/package-release.sh`** — builds a `process-monitor-manager-v<version>.tar.gz` straight from the git tag via `git archive`, so it only ever contains tracked files.
- **`scripts/build-deb.sh`** — stages a `usr/bin` wrapper and a `usr/lib/process-monitor-manager/` copy of the scripts, generates a `DEBIAN/control` file with the right dependency list, and builds a `process-monitor-manager_<version>_all.deb` via `dpkg-deb` — installable with `sudo apt install ./process-monitor-manager_<version>_all.deb`, which puts a `process-monitor-manager` command on the `PATH`.

`.github/workflows/release.yml` runs both scripts and attaches the resulting `.tar.gz` and `.deb` to a GitHub Release whenever a `v*` tag is pushed — this is the direct answer to "there's no `.exe`, so what do we ship instead?"

## 🔄 Changed

### ⚠️ CLI Menu Renumbered (Breaking)

`monitor.sh`'s menu gained a "9. Export to CSV" option ahead of Exit, which moved from `9` to `10`. Anything that scripted or muscle-memorized `9` to exit the CLI will now trigger a CSV export instead of exiting — this is the reason for the major version bump rather than a minor one, even though nothing else in this release is behavior-breaking.

---

### 🖥️ GUI Radiolist

`monitor_gui.sh` gained an "Export to CSV" action alongside the existing seven.

## 🏛️ Architecture Progress

**Configuration via environment, not files.** Signal deliberately avoids introducing a config-file parser for two threshold numbers — `PMM_HIGH_THRESHOLD`/`PMM_MED_THRESHOLD` env vars are read identically by the Bash modules (`${PMM_HIGH_THRESHOLD:-50}`) and by `dashboard.py` (`os.environ.get(...)`), keeping the "no third-party dependencies" property from Pulse intact.

**Release artifacts built from git, not from the working tree.** `package-release.sh` uses `git archive` against the pushed tag rather than tarring the checkout directly, so a dirty working tree or stray local files can never leak into a release artifact — the tarball is exactly what's committed.

Known limitations at this release:

* No config-file option for thresholds — environment variables only.
* No `.rpm`/Homebrew/Snap packaging — `.deb` and a plain tarball cover Debian-based distros only (see [ROADMAP.md](../../ROADMAP.md)).
* The `.deb`'s `usr/bin` wrapper always `cd`s into `/usr/lib/process-monitor-manager` before running, since `monitor.sh` sources its modules with paths relative to the current directory rather than its own script location.
