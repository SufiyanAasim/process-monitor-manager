# Process Monitor & Manager — v5.0.0

> *Codename: Beacon*

---

## 📖 Overview

Beacon combines two things that turned out to belong in the same release: a deep, line-by-line bug and consistency audit of everything shipped since Bedrock, and the first round of polish features — a Credits screen, persistent config-file thresholds, and a proper app icon in the GUI. Neither half is glamorous alone; together they're the point where the project stops being "a script that works" and starts being "a tool someone else would trust."

Building on Signal's packaged-release pipeline, this release also completes the "inserting an option ahead of Exit" pattern started in Signal — Credits was added at position 10, pushing Exit to 11, which is why this is `v5.0.0` and not a patch release layered on top of the bug fixes.

---

## 🎯 Objectives

This milestone was developed with the following goals:

* Audit every shell and Python file for correctness bugs, error-handling gaps, and interface inconsistencies — not just the ones exercised by existing tests.
* Make the tool resilient to bad input instead of forwarding it straight to `kill`.
* Give the project a face: a Credits screen, a usable app icon, and a way to persist configuration instead of re-typing environment variables every run.
* Close the gap between "works when I run it a specific way" and "works."

## ✨ Added

### 👤 Credits (`@author: Sufiyan Aasim`)

CLI option 10 and a GUI "Credits" action, listing both members with their role, GitHub handle, and contact email.

Each member gets their own button in the GUI — **GitHub: @SufiyanAasim** and **GitHub: @13eeCoder** — rather than one shared repository link, so each profile is reachable directly. This uses Zenity's `--extra-button`, whose contract is that OK exits 0, an extra button exits 1 while printing its own label, and Cancel exits 1 printing nothing; that's what distinguishes the three outcomes.

Opening a URL goes through a small `open_url()` helper that tries `xdg-open`, then `wslview`, then falls back to displaying the URL. `wslview` (from `wslu`) is in the chain because this tool is commonly run under WSL, where `xdg-open` typically isn't installed but a Windows browser is still reachable — without it, the button would dead-end for exactly the environment the project documents most heavily.

### ⚙️ Config-File Support for Thresholds (`@author: Sufiyan Aasim`)

`PMM_HIGH_THRESHOLD`/`PMM_MED_THRESHOLD` can now be set persistently in `~/.config/process-monitor-manager/config` (`$XDG_CONFIG_HOME` respected) instead of exporting an environment variable on every run. An explicit environment variable still overrides the config file. See `config/process-monitor-manager.example` for the format. The Bash modules and `dashboard.py` read the config file the same way, with the same precedence.

```bash
mkdir -p ~/.config/process-monitor-manager
cp config/process-monitor-manager.example ~/.config/process-monitor-manager/config
```

### 🖼️ App Icon in the GUI (`@author: Sufiyan Aasim`)

A new `assets/icon.svg` is passed to every Zenity dialog via `--icon`, so the project mark shows in the window/taskbar instead of a generic one.

It's deliberately a *second* asset rather than reusing `assets/logo.svg`. GTK renders a dialog icon at roughly 48px, and at that size the logo's 9-unit stroke in a 240-unit viewBox rasterizes to about 1.8px, with its traffic-light dots landing around 4px across — the detail that makes the logo work at README size is exactly what turns it to mush at icon size. `icon.svg` drops the window chrome and dots entirely and carries a 6-unit stroke in a 64-unit viewBox (~4.5px at 48px, 2.5× bolder), keeping just the pulse mark. It's viewBox-only, with no fixed `width`/`height`, so it scales to whatever size a consumer asks for.

`monitor_gui.sh` prefers `icon.svg`, falls back to `logo.svg`, and finally to no icon at all — a partial install degrades instead of erroring. Both assets are now included in the `.deb` package staging in `scripts/build-deb.sh`, which previously omitted `assets/` entirely, meaning an installed `.deb` build would have silently shown no icon regardless.

### 🔙 Back Buttons Throughout the GUI (`@author: Sufiyan Aasim`)

Sub-dialogs — the Kill/Suspend/Resume process pickers, the search prompt, and the Credits screen — now label their dismiss button **Back** rather than "Cancel". They return you to the main menu, which "Cancel" implied was aborting something. The main menu's own cancel is labelled **Exit**, since that's what it does.

Read-only reports (process info, search results, alerts) render through a new shared `show_report()` helper. Both of their buttons read "Back": Zenity rejects `--no-cancel` for `--text-info` ("not supported for this dialog"), so collapsing them into one button isn't available — and since both genuinely do the same thing, labelling them identically is at least accurate.

`tests/smoke_test.sh` verifies Zenity actually *honours* these flags rather than silently dropping them. That check exists because Zenity's failure mode here is quiet: given an unsupported flag it prints a note to stderr and renders the dialog anyway, so a Back button that never appeared would look identical to a working one from the script's side.

## 🐛 Fixed (`@author: Sufiyan Aasim`)

This release folds in a full hardening pass, found by re-reading every script line by line rather than waiting for a bug report (`@author: Sufiyan Aasim`):

- **Executable bits were never committed to git.** `core.filemode=false` in the development environment silently dropped every `chmod +x` at staging time — every prior commit tracked every script as `100644` (non-executable). A fresh `git clone` on Linux needed a manual `chmod +x`, and the CI release workflow's direct `./scripts/*.sh` calls would have failed outright. Fixed by explicitly setting the executable bit in the git index (`@author: Sufiyan Aasim`).
- **`monitor.sh` broke when invoked from outside the repo root** — the same class of bug `v2.0.0` fixed, just relocated from a missing `modules/` folder to a missing `${BASH_SOURCE[0]}` reference. Paths now resolve relative to the script's own location (`@author: Sufiyan Aasim`).
- **`monitor_gui.sh`'s "Kill a Process" list rendered as one merged column** instead of two, because `--column="PID:Command"` is a single string, not two `--column` flags. Replaced with a real two-column, `--print-column=1` list (`@author: Sufiyan Aasim`).
- **Live Dashboard PID search required an exact match**, unlike the CLI/GUI's substring match. Now consistent (`@author: Muhammad Taha Siddiqui`).
- **No input validation before signaling a process.** Empty, non-numeric, or self-referential PIDs (the tool's own shell/process) reached `kill` directly — a self-targeted `SIGSTOP` would freeze the session with no in-session recovery. Kill/Suspend/Resume now validate across the CLI, GUI, and Live Dashboard (`@author: Muhammad Taha Siddiqui`).
- **Search matched as regex, not literal text** (`grep -i` → `grep -iF`), so a query like `a.b` matched `aXb` unintentionally (`@author: Sufiyan Aasim`).
- **`colorize_line` could throw a bash arithmetic error** on a malformed `%CPU` value; sanitized to digits-only first (`@author: Sufiyan Aasim`).
- **`--window-icon` is deprecated in current Zenity** — this release's own icon work uses `--icon` instead, the maintained flag, avoiding shipping a deprecated integration on day one (`@author: Sufiyan Aasim`).
- **Any `&`, `<` or `>` in an interpolated dialog silently aborted that dialog.** Zenity parses `--text` as Pango markup by default, so a checkout under a path like `.../Fully Tested & Deployed/...` made the "no terminal emulator" error and the CSV-export confirmation fail with `Failed to set text ... from markup due to error parsing markup` instead of displaying — as did any search query containing a markup metacharacter. Every value-interpolating dialog now passes `--no-markup`; only the static Credits text (with `&` pre-escaped as `&amp;`) still opts in (`@author: Sufiyan Aasim`).
- **…and that fix had a tail.** `--no-markup` quietly disables `\n` handling too, so the newly-visible dialogs printed a literal `\n` on screen. Multi-line `--no-markup` text now carries real newlines. Both properties are test-guarded, since each one fails invisibly: markup-on corrupts paths, and markup-off corrupts escapes (`@author: Sufiyan Aasim`).
- **The GUI's window close (X) button did nothing.** Cancelling or closing the main radiolist fell through to a `*)` branch running `continue`, which immediately reopened the same dialog — the only way out was selecting "Exit" from the list itself. Cancel and X now exit, and the button is labelled "Exit" to say so (`@author: Sufiyan Aasim`).

### 📤 The Export-to-CSV Cluster (`@author: Sufiyan Aasim`)

Three separate defects that combined into one: **every `.deb` install had CSV export silently broken, on every attempt.**

1. `export_csv()` never checked whether its output redirection succeeded. In an unwritable directory it returned exit 0 and printed a filename regardless — so the CLI printed "Exported current process snapshot to …" and the GUI showed a success dialog, with nothing on disk.
2. The packaged `usr/bin` wrapper ran `cd /usr/lib/process-monitor-manager` first, dropping every user into a **root-owned** directory — precisely the unwritable case above. That `cd` existed to work around the relative-module-sourcing bug *this same release fixes*, making it simultaneously obsolete and actively harmful.
3. `dashboard.py`'s `export_csv()` let `PermissionError` escape the curses loop, so the same condition took the entire TUI down rather than showing a message.

All three are fixed: the Bash side returns non-zero and prints nothing on failure, the wrapper `exec`s an absolute path so the CSV lands in the user's own directory, and the Python side returns `None` for the dashboard to report inline.

The first attempt at fix (1) was itself wrong, and worth recording: `if ! { …; } > "$outfile"` **never fires**. When the redirection itself fails, bash returns 1 for the whole construct and the `!` does not invert it — so the guard silently never triggered and the test caught it. It's now an `if`/`else` with no negation.

### 🖥️ Live Dashboard Crashes (`@author: Sufiyan Aasim`)

- **Resizing the terminal killed it.** `draw()` sized its output from `curses.LINES`/`curses.COLS`, which are captured at `initscr` and go stale on resize. Measured directly: after shrinking 24 → 10 rows, `curses.LINES` still reported 24 while `stdscr.getmaxyx()` correctly reported 10 — so the row loop wrote past the new last line and raised `curses.error`, unwinding the whole TUI. Every write now clips against a live `getmaxyx()`, guards terminals shorter than the header, and swallows `curses.error` as a backstop.
- **The GUI couldn't launch it on a standard Ubuntu desktop.** The launcher ran `$TERMINAL -e python3 <path>` — three separate arguments. `xterm` and `konsole` accept that; `gnome-terminal` and `xfce4-terminal` take only a single string after `-e`, leaving the path as a stray argument. And `x-terminal-emulator`, the first candidate tried, is normally a symlink to `gnome-terminal` on Ubuntu — the project's primary target. Since the launch is backgrounded, the error never surfaced in a dialog. The launcher now resolves `x-terminal-emulator` to its real target and passes each terminal's documented flag (`--`, `-x`, or `-e`).
- A documentation-consistency pass on top of the code fixes caught: a twice-wrong `PMM_MED_THRESHOLD` description, a stale `Architecture.md`, missing Troubleshooting/FAQ coverage for the new validation messages, a `SECURITY.md` supported-versions table stuck on `3.x`, a PR template referencing pre-`v3.0.0` commands, and several commit/branch examples describing already-shipped work as pending.
- CI's `bash -n` check omitted `scripts/*.sh`/`tests/*.sh`, and `release.yml` was missing `permissions: contents: write` (would 403 on restrictive default token permissions).

## 🏛️ Architecture Progress

**Zenity dialogs now share common `ICON_ARGS` and `NO_MARKUP` arrays**, built once at the top of `monitor_gui.sh` and splatted into every `zenity` call. `ICON_ARGS` resolves to an empty array if neither icon asset is found, so a stripped-down install degrades gracefully instead of erroring. `NO_MARKUP` makes markup-off the default posture and markup-on the single, commented exception, rather than the reverse — the markup bug this release fixes was possible precisely because Zenity's default is the unsafe one.

**The markup rule is enforced by a test, not a convention.** `tests/smoke_test.sh` joins line-continuations, extracts each `zenity` invocation individually, and fails if any lacks `--no-markup` without being the Credits dialog. Extracting per-invocation rather than per-line is load-bearing: the first version of this check ran per-line and silently passed a deliberately-broken dialog, because one safe call in an `a && b || c` chain masked its unsafe sibling on the same source line.

**Config resolution is symmetric across runtimes.** Both `monitor_module.sh` (Bash) and `dashboard.py` (Python) implement the same three-tier precedence — environment variable, then config file, then hardcoded default — independently, since the two runtimes don't share code today. The Bash side captures any pre-set environment variables before sourcing the config file, then re-applies them afterward, so the config file can never silently override an explicit `PMM_HIGH_THRESHOLD=X ./monitor.sh` invocation.

Known limitations at this release:

* Credits is CLI/GUI only — the Live Dashboard's curses interface doesn't have a Credits key, since "clickable button" doesn't map cleanly onto a TUI.
* The config file has no schema validation — a malformed line is silently ignored rather than erroring, falling through to the next precedence tier.
