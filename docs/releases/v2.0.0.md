# Process Monitor & Manager — v2.0.0

> *Codename: Bedrock*

---

## 📖 Overview

Bedrock turns the Spark scaffold into a working baseline. No new user-facing features were added — this release is entirely about making the existing CLI and GUI actually run, and confirming they do, on a real Ubuntu environment, before any further feature work begins.

Building on the module-per-responsibility shape established in **Spark**, this release focuses on closing the gap between what the code intended to do and what it actually did when someone tried to run it.

---

## 🎯 Objectives

This milestone was developed with the following goals:

* Diagnose why CLI mode failed immediately on a fresh clone.
* Fix the packaging mistake without changing any behavior.
* Verify both CLI and GUI mode end-to-end on a real Ubuntu environment.
* Establish a trustworthy baseline before adding any new features.

---

## 🐛 Fixed (`@author: Sufiyan Aasim`)

### 📁 Broken Module Path (`@author: Sufiyan Aasim`)

`monitor.sh` sourced:
```bash
source modules/monitor_module.sh
source modules/manage_module.sh
source modules/input_module.sh
```
but the repository shipped `monitor_module`, `manage_module`, and `input_module` at the root — no `modules/` directory, no `.sh` extension. CLI mode failed immediately with a "No such file or directory" error.

Moved all three scripts into `modules/`, renamed with the `.sh` extension, and made them executable alongside `monitor.sh` and `monitor_gui.sh`.

## ✅ Verified (`@author: Sufiyan Aasim`)

### 💻 CLI Mode (`@author: Sufiyan Aasim`)

Menu loop, Show Process Info, Show Process Tree, and Kill/Suspend/Resume confirmed working end-to-end on Ubuntu via WSL.

---

### 🖥️ GUI Mode

Zenity dialogs confirmed rendering and functioning via [WSLg](https://github.com/microsoft/wslg), including the full action set inherited from Spark.

---

## 🏛️ Architecture Progress

**No architectural change — a correction, not a redesign.** The module-per-responsibility split from Spark was already the intended design; Bedrock only makes the filesystem match that intent. No source logic changed, which is why this release carries no `### Added` or `### Changed` sections of its own.

Known limitations carried over from Spark:

* No search, filtering, alerting, or live/auto-refreshing view.
* No automated tests, CI, or contributor documentation.
